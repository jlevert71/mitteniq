# MittenIQ Roadmap

Last updated: 2026-03-02

---

## Core Purpose (Confirmed)

MittenIQ exists to reduce the time and cost required to prepare construction estimates by removing the minutiae involved in setting up an estimate.

The system is intended to:
- Serve as an accurate and trustworthy resource for examining project specifications.
- Save estimator time so effort can focus on higher-value work.
- Reduce operational cost compared to traditional estimating workflows.
- Remain simple, reliable, and visually clean with a modern contractor-focused design.

Primary problem addressed:
Time inefficiency and cost overhead in estimate preparation.

---

## Phase 1 — Project Intake & Organization (Confirmed)

### Goal
Provide a complete, trustworthy project setup workflow that removes manual effort required before estimating begins.

### Primary Workflow
1. User creates a project.
2. User uploads construction documents.
3. System performs intake analysis.
4. System creates a logical, exportable project folder architecture.
5. Documents are automatically sorted into folders.
6. Original complete document set is preserved unchanged.
7. Intake produces a report describing what occurred.

### Value Delivered
- Eliminates setup minutiae before estimating.
- Produces structured, ready-to-use project documentation.
- Provides confidence that project inputs are organized and traceable.

### Success Criteria
A user uploads files and receives a fully organized, clearly reported project setup without manual sorting.

## Phase 1 Design Specification — Project Folder Architecture (Confirmed Concept)

During Intake, MittenIQ creates a standardized, exportable project folder structure.

Rules:
- Only folders relevant to detected documents are created.
- New disciplines or specification sections are added dynamically if encountered.
- Original documents are never modified or removed.

### 05_Addenda

Created when addenda exist or are later introduced.

Files named using addendum number and date.

Intake parses addenda to identify:
- scope changes
- specification amendments
- drawing revisions
- written drawing changes

#### Addendum Document Handling Rules (Confirmed Design Intent)

When addenda introduce revised drawings or specifications:

**Drawing Handling**
- Original drawings are NEVER deleted.
- Revised drawings stored as addendum versions.
- Naming reflects addendum source (example: `E1 - Addendum 01`).
- Revised drawings appear in ALL relevant locations:
  - updated within the complete project set
  - updated within the discipline folder
  - stored within the Addenda folder

**Specification Handling**
- Reissued specifications preserved as addendum versions.
- Files labeled with addendum origin.
- Original versions remain preserved.

**System Requirement**
MittenIQ maintains a historical chain showing:
- original issue
- addendum revision
- latest active version

Goal:
Users always know the current version while retaining full history.

### 06_Bid Forms and Reports

Extracted from specifications into standalone files.

Includes:
- Bid Forms
- Instructions to Bidders
- Project Electrical Related Scopes Report (generated after analysis)

The Project Electrical Related Scopes Report:
- is generated by MittenIQ after document analysis and separation
- is not a static extracted file
- must automatically update when addenda or new information are introduced
- reflects current project scope based on latest documents
